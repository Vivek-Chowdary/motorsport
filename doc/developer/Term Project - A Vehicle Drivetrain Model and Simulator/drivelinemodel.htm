1.    Introduction

1.1.        Problem

The University of Guelph Formula SAE car was the first all-wheel-drive (AWD) car to compete at the annual FSAE competition in May, 2004.  However, it was not judged favorably because it lacked the data to prove that it is more competitive than an equivalent lighter rear-wheel-drive (RWD) configuration.  The model and simulation developed in this project will be used to compare the performance characteristics of the two systems.

1.2.        Original Design Constraints

·        The project must be completed by mid December, 2004.

·        There is no budget allotted for development.

1.3.        Original Design Criteria

·        Flexibility  can be used to analyze other parameters

·        Robustness  can add more detail as parameters become available

·        User friendliness  understandable with some mechanics and dynamics background

2.    Background

2.1.        University of Guelph Formula SAE 2004 Drivetrain

The 2004 Gryphon Racer was designed with an innovative AWD System designed to be an add-on to a permanent rear-wheel-drive(RWD) vehicle as shown in Figure 1.  In this case the rear wheels are continuously driven and torque is transferred via a controlled coupling to the front wheels only when the front and rear angular velocities are different.  This coupling is an electronically controlled, hydraulic clutch mechanism manufactured by Haldex Traction Systems.  The front and rear differentials are both from the front differential of a 2002 Suzuki Vinson 500 all-terrain vehicle.  The engine is from a 2002 Suzuki GSX-R600 sport motorcycle.

Figure 1 - 2004 Gryphon Racer Drivetrain View

2.2.        Open Dynamics Engine

The Open Dynamics Engine (ODE) is an open source library designed to simulate the articulation of rigid bodies connected by joints and the associated dynamics.  It includes a C++ application programming interface that is platform independent.

2.3.        Motorsport Vehicle Simulator

Motorsport [motorsport-sim.org] is an online project with the goal of developing an open source real-time vehicle simulator.  The project is multi-platform and is being coded in C++.  It uses the libraries from ODE for dynamics, OGRE(object oriented graphics rendering engine) for graphics, SDL(simple direct-media layer) for input, and the XML data format for configuration files.

3.1.        Basic Drivetrain Model

3.1.1.         Layout

The basic rear wheel drive(RWD) layout was developed as a starting point for modelling, as shown in Figure 2.  The inertia (J) and friction(B) of each rotating mass is included, and the gears are modeled as transformers.

Figure 2  Oriented Linear Graph for Basic RWD Drivetrain

3.1.2.         Assumptions

·        driven wheels linked by solid axle, so resulting tire torques are identical (no lateral motion)

·        neglecting changes in gearbox inertia as different gears are selected

·        perfectly flat engine torque curve (independent of engine speed)

·        clutch modeled as friction between engine and gearbox

·        no compliance or backlash in components

·        purely rotational motion in engine (no reciprocating parts)

3.1.3.         System Response

The wheel speed is the most important output of the system, since the dynamics of the tire with the road define the resulting torque back into the system.  Equation 1 is the simplification of the 28 system equations:

Equation 1  Wheel Speed Response of System

The response is 2nd order since there are only two independent storage elements if Rd, Rg, and Bcl are not zero.  If the clutch is assumed to be fully engaged(locked), Bcl becomes infinite and the system reduces to 1st order as shown in Equation 2.

 

Equation 2  Wheel Speed Response of System with Locked Clutch

At any given time Rg can become zero when neutral is selected, which presents a problem with implementation since there is neutral state between any gearshift operation.  Also, if a gear is selected and the corresponding velocities are not matched, an infinite acceleration is imparted on the component.

 

For an implementation of this model, equations would need to be developed and coded for each variation (RWD, FWD, 4WD, front-bias AWD, rear-bias AWD), which would be very time consuming.


3.2.        Reconfigurable Model

3.2.1.         Layout

In order for the model to be easily reconfigured, it was broken into sections as shown in Figure 3.  The separations are not necessarily where the velocity changes from a gear but where the system needs to be isolated.  For example, the ring gear in the differential is rotating at a different velocity, but is lumped with the final drive because it is completely dependent on the velocity of that mass.  The torque split for the wheels is inside the differential, so that is where the joint would connect. 

 

Figure 3  Oriented Linear Graph of Basic Model Broken Into Sections

 

A spring was added between each mass to keep their rotational velocities independent when solving the system (Figure 4).  A damper was also included to keep the system stable.  The output velocity in a geared system is contained within the mass with a defined ratio to the input velocity and joint torque from the output side is divided by the same ratio when solving.

3.4.4.1.                Gear Joint (Fixed Joint)

This joint connects two masses with a spring and damper combination.  The constants are tuned to reduce the amount of flex in the system, however the limit of tuning is the time constant of the system, which must be greater than twice the calculation step size to retain stability.

3.4.4.2.                Clutch

The clutch is modeled as pure frictional element between two components.  There is a maximum torque transfer parameter that corresponds to the maximum static friction of the clutch plates.

3.4.4.3.                Limited Slip Differential (LSD)

This joint implementation requires a second output as a special case.  The code will be developed to properly model torque biasing properties of open, locked and limited-slip differentials.  It can be used to drive two wheels or as a transfer case differential in a 4WD configuration.

 

4.    Discussion

4.1.        Simulator Usage

At this point, an AWD drivetrain can be created to measure performance differences between the two systems.  Output for vehicle position vs time can be analyzed with the telemetry streamed to log files and parsed. 

The system also has the ability to test the vehicle performance gain with the reduction of rotating mass.  This is useful for engine development as test cases can be generated for reducing the crankshaft mass, as well as removing higher gears that are not needed.

4.2.        System Stability

Relatively large spring constants are required to accurately simulate the torque transfer between two masses that are essentially directly connected.  This leads to problems where small oscillations become damped large oscillations when simulated.  This is called a stiff problem. Reductions in the time step lead to accumulation of rounding errors, but increasing the time step causes stability problems.  The solution is to find the best integration method that has a low error but is not computationally expensive. 

4.3.        Simulator Limitations

The Motorsport simulator is still in its infancy; however it is close to Evolution1 release for general testing.  The code is being optimized and there are a few issues with the validity of any data obtained.  The tire model is a linear friction model which does not account for slip angles or slip ratios of the wheels, which is critical for wheelspin and launch characteristics.  The suspension is also very basic, so lateral and longitudinal load transfers in transient and steady state will be affected.  The engine is also modeled as a flat torque curve, independent of the engine speed, which affects the proper selection of gears at high rpms.

4.4.        Further Research

The drivetrain model has the potential to be a very useful tool if developed properly.  The friction parameters need to be modeled including the engine friction, and the clutch model should be improved.  Once the stability issues are resolved, the actual stiffness, compliance, and backlash of components can be calculated and simulated in the joints.
